Code Documentation Best Practices

Learn how to write clear, maintainable comments and documentation that help your team understand and maintain your codebase effectively.

Why Documentation Matters

Good documentation is essential for maintaining codebases over time. It helps new team members understand the code, explains complex logic, and documents important decisions.

Writing Effective Comments

✓ Good Comment Practice

  • Explain why, not what
  • Keep comments up-to-date with code changes
  • Use comments to explain complex algorithms
  • Document assumptions and limitations
// Good: Explains why we use a delay // Adding 100ms delay to prevent API rate limiting (max 10 req/sec) await sleep(100); // Bad: States the obvious // This adds 100 milliseconds to the wait time await sleep(100);

HTML/XML Comments

In markup languages, comments can document structure and provide context for designers and developers:

<!-- Navigation Section --> <nav class="main-navigation"> <!-- Primary menu items --> </nav> <!-- Hero Section: Update banner image seasonally --> <section class="hero"> </section>

Documentation Comments (JSDoc, JavaDoc, etc.)

Use structured documentation comments for functions, classes, and modules:

/** * Calculates the total price including tax and shipping * @param {number} basePrice - The base price before additions * @param {number} taxRate - Tax rate as decimal (e.g., 0.08 for 8%) * @param {number} shippingCost - Fixed shipping cost * @returns {number} Total price with tax and shipping */ function calculateTotal(basePrice, taxRate, shippingCost) { return basePrice * (1 + taxRate) + shippingCost; }

Common Pitfalls to Avoid

  • Commenting every single line (over-documentation)
  • Leaving outdated comments that contradict the code
  • Using comments to "comment out" old code (use version control instead)
  • Writing vague comments that don't add value
  • Forgetting to update comments when refactoring code
